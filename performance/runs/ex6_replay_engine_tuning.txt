ex6 replay — engine micro-tuning after the O(log) BBO migration
===============================================================

What this is
------------
Picks up where ex6_replay_bbo_after.txt left off. That report ended with the
engine dominated by its own data-structure maintenance and flagged the next
targets. This documents four follow-up optimizations found by re-profiling,
all on the PR-#19 nested book (price -> hash-queue). Same driver, preset, seed.

  branch : robyn/ex6-workload-profile
  cmd    : perf record -F 999 -- main.exe replay -preset balanced \
             -num-actions 1000000 -depth-every 0
  raw    : perf-after2.data (before the level cache) / perf-after3.data (after)
           both gitignored

The four changes
----------------
1. Reverse lookup table removed (commit "carry client_order_id on Order.t").
   Order.create discarded the client_order_id, so the engine kept a side
   hashtable (order_id -> client_order_id) purely to relabel fills. Now Order.t
   carries the id and fills read it as a field. -2 reverse-table lookups per
   fill.

2. Per-level size cache (this commit). best_bid_offer computed the top-of-book
   size by folding the whole best-price queue on every read (level_size ->
   Order_queue.fold). Each price level now carries a mutable total_size,
   maintained at three sites — add (+), remove' (-), and fill_resting (-).
   best_level reads it in O(1). The subtlety: Order.fill mutates a resting
   order's remaining_size WITHOUT going through the book, so resting-order fills
   in match_loop now route through Order_book.fill_resting, which shrinks the
   order and the cached level total together. A bare Order.fill on a resting
   order would silently rot the cache.

3. Duplicate client-order-id check collapsed (this commit). submit did
   Hashtbl.mem (in validate_client_id) and then Hashtbl.add_exn (in
   add_client_order) — two probes of the same inner table per submit. Now one
   Hashtbl.add returns [`Ok | `Duplicate]; the common path probes once.

4. Incremental top-of-book cache (this commit). best_bid_offer and find_match
   still walked the price map (Map.max_elt/min_elt, O(log P)) on every read.
   The book now caches best_bid / best_ask (the top level per side) and reads
   them in O(1). Maintained incrementally: an add whose price is strictly more
   aggressive than the current best becomes the new best (O(1)); removing the
   last order at the best level recomputes via the O(log P) walk — the only
   path that still walks. A resting fill just shrinks that level's total_size
   in place, so the cached pointer needs no update. Bonus: Map.max_elt/min_elt
   allocate a (key,data) option on every call, so caching also cut allocation.

Evidence: read the PROPORTIONS, not the wall
--------------------------------------------
The two captures ran at very different box contention (perf sample duration:
29.5 s for after2, 3.9 s for after3 — another session was building during the
first). So absolute wall and latency are NOT a clean comparison. The reliable
signal is the self-normalized self-time proportions, which show the mechanism
worked:

  symbol                         after2   after3   what happened
  ------------------------------------------------------------------
  Hash_queue.foldi (level fold)   4.42%   0% (gone) best_level stopped folding
  Doubly_linked.with_iteration    2.57%   0% (gone) the fold's DLL traversal
  Hashtbl.mem                     5.28%   1.97%     dropped the redundant probe
  best_level_entry (BBO read)     5.50%   4.48%     now just the Map min/max walk
  Hashtbl.find_and_call           9.30%  10.18%     now #1 — bigger slice, smaller pie
  allocation                    141.6 w   140.8 w   flat: this is a CPU win, not memory

The fold symbols left the profile entirely (zero samples in after3), which is
the point: an O(k)-per-read fold became an O(1) field read plus O(1)-per-write
maintenance, and BBO is read twice per submit and twice per cancel while each
order only enqueues/dequeues once — so moving the work to the write side is a
clear net win.

Absolute numbers: the cumulative journey (O(n) -> fully tuned)
-------------------------------------------------------------
Two kinds of number here. ALLOCATION and GC COUNTS are deterministic (same seed
=> same work every run), so they compare exactly across any run. WALL and
LATENCY depend on box contention, so those come from a back-to-back run of the
two binaries one right after the other at load ~3-5 both times -- and the tuned
leg actually ran at the HIGHER load (5.15 vs 2.82), so its advantage is
understated, not inflated.

Deterministic (1M actions, contention-independent):
  state                          alloc w/action   minor GC   major GC
  O(n) baseline (pre-PR#19)          3049.9          2957        86
  + PR#19  O(log) BBO                 145.9           158        29
  + reverse-table removal             141.6           153        28
  + level-cache + dup-check           140.8           152        28
  + BBO cache (now)                   124.6           136        26
  => 24.5x less allocation, 21.7x fewer minor GCs, cumulative from O(n).

Matched back-to-back (1M actions, same box, consecutive):
                        O(n) baseline    fully tuned    factor
  wall                    47.16 s          4.57 s        10.3x
  throughput              21.2k act/s      218.9k act/s  10.3x
  submit p50              65.54 us          2.05 us       32x
  submit p99            524.29 us          16.38 us       32x
  cancel p99           131.07 us           8.19 us       16x
  events                 identical         identical     (correctness)

Where the wins came from (submit p50, the headline latency):
  O(n) 65.54us -> PR#19 4.10us (16x, the big structural win: no more O(n)
  list-fold BBO) -> reverse-table 4.10us (flat here; that win was on the fill
  path) -> level-cache + dup-check 2.05us (another 2x). PR#19 did the heavy
  lifting on allocation and latency; this session's three tweaks then doubled
  submit again on top by deleting the residual per-read fold and the redundant
  hashtable probes -- CPU-cycle wins, which is why they barely move allocation.

BBO cache isolated (pre vs post, matched back-to-back, same low load)
--------------------------------------------------------------------
The level-cache+dup-check binary vs the BBO-cache binary, run consecutively:
  best_level_entry (the Map min/max walk)   4.48%  ->  0.09%  (off the profile)
  allocation                               140.8   ->  124.6 w/action  (-11.5%)
  minor collections                          152   ->  136
  throughput                              396k     ->  430k act/s      (+8.5%)
  submit p50                               2.05 us ->  2.05 us   (unchanged: the
    saved O(log P) walk is below the histogram's ~us resolution; the win shows
    up as throughput + allocation, not median latency)
The allocation drop is the interesting part: Map.max_elt/min_elt each allocate
a (key,data) option, and we called them ~8x per action on the read path (2 per
best_bid_offer x 2 reads/submit + 2/cancel, plus find_match). The cache returns
a stored option, allocating nothing, so killing the walk also killed its
garbage. Diminishing returns, though: only ~8% throughput, because the engine
is now a small enough slice of the whole that Amdahl caps the win.

What's next
-----------
find_and_call is now the single biggest cost (11.30%) — order-id hashtable
lookups (id_hash on add/remove/find) plus the client_order_tables (get on
cancel, find_or_add on submit). best_level_entry has left the profile (0.09%).
GC is ~18% of this (much smaller) profile and the workload generator ~7% and
rising: we are near the point where the RNG, not the engine, is what's left to
optimize.
