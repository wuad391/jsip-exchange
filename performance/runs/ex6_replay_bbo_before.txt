ex6 replay profile — BEFORE (O(n) best-bid-offer recomputation)
================================================================

What this is
------------
`perf` self-time profile of the Ex6 replay driver running one workload
through the matching engine, on the *un-optimized* order book (the
list-backed `best_level` that materializes the whole book on every read).

  branch : robyn/ex6-workload-profile (based on aba356f, pre-PR-#19)
  binary : performance/bin/main.exe   (preserved: $CLAUDE_JOB_DIR/tmp/baseline-main.exe)
  cmd    : perf record -F 999 -- main.exe replay -preset balanced \
             -num-actions 1000000 -depth-every 0
  raw    : perf-balanced-before.data  (worktree root; symbols only resolve
           against the baseline binary above — decode kept here as text)
  samples: 28K task-clock, sample duration 33.8 s (box heavily contended;
           SELF-TIME PROPORTIONS are trustworthy, absolute wall time is not)

Headline
--------
~75-85% of the whole program is best-bid-offer recomputation. Every submit
AND every cancel calls best_bid_offer, which calls best_level for both sides,
which does `List.map (Map.to_alist book)` + `List.fold` over EVERY resting
order (~3000 at steady state) to sum the top level. That is the O(n)-per-write
BBO the order-book bench flagged. PR #19 replaces it with a nested
price -> hash-queue book whose best_level is Map.min_elt = O(log P + k).

Note on GC: the collector is only ~3% despite 1574 words/action allocated,
because nearly all of those words are the transient to_alist lists + Level
records that die in the minor heap before promotion — cheap bump-pointer
minor collection, little marking. (Earlier I mis-grepped for pre-OCaml-5 GC
symbol names and wrongly said GC was ~0; the OCaml-5 collector symbols are
oldify_one / do_some_marking / pool_sweep, and they sum to ~3%.)

THE O(n) TRAP — cost scales with book depth. Compare 200k vs 1M on the SAME
un-optimized binary:
  200k:  submit p50=16.38us,  1574.7 words/action,  minor=313
  1M:    submit p50=65.54us,  3049.9 words/action,  minor=2957
Per-action cost roughly DOUBLES from 200k to 1M, because best_level's
`List.map (Map.to_alist book)` allocates a list proportional to the current
book size, and a larger fraction of the 1M run happens at the full ~3000-order
plateau. The longer you run, the worse O(n)-per-write looks. The optimized
book (ex6_replay_bbo_after.txt) is flat: 145.9 words/action at BOTH scales.

    150k:2872 200k:3024 -> ~3k plateau
