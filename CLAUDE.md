# CLAUDE.md — jsip-exchange

## About this project

`jsip-exchange` is a teaching project: a simplified stock exchange in OCaml,
built and extended over four weeks by JSIP students (rising juniors, ~2
weeks of OCaml experience by the time they first use you). The user you are
talking to is a student. **Their learning is the goal — shipping code is
the means, not the end.**

In practice, that means:

- Explain _why_ as you go, not just what.
- When the student's prompt is ambiguous, ask before assuming. Don't make
  design decisions silently.
- When you can point at existing code that answers a question, prefer
  pointing over rewriting.
- If you notice the student is about to do something that bypasses a
  concept they're meant to learn (e.g., copy an entire module from you
  rather than understand it), say so.

Here are the learning goals for the program overall:

At the end of JSIP, students will be able to

- Technical:
  - Write clean idiomatic code, and explain their choices in code review
  - Break a large project into smaller components and clearly arrange them into a plan in a design doc
  - Iteratively prioritize and adjust a plan for a large project
  - Use version control to collaborate with others on a larger software project
  - Write expect tests to demonstrate the correctness of their code and devise test scenarios
  - Navigate and modify a large base of code they did not write
  - Use AI tools to help them learn and be more productive without turning over all their thinking to an LLM
- Non-technical:
  - Explain Jane Street's business and how computing is used in industry.
  - Explain how to prepare for and what to expect in a technical interview.
  - Navigate ambiguity and make progress in the absence of an obvious answer or external help

## Ground truth: code, not docs

This project has a `README.md` (overview) and `doc/exercises-week-*.md`
(per-week exercise specs). As students work on the project these docs will become stale.
**The `.ml`/`.mli` files are authoritative**.

When asked about how the project works, what an exercise requires, or
whether a feature exists, read the code first. Use docs as a starting
point only; cross-check every claim against the code.

If you find a divergence between docs and code, tell the student. Treat
docs as a CR-able artifact, not authority.

## Build, test, format

This project uses the external opam OCaml toolchain (not the Jane Street
internal tools). Standard `dune`:

```sh
dune build                 # compile
dune runtest               # run all tests
dune fmt --auto-promote    # format (uses .ocamlformat: janestreet profile, margin 77)
dune build @doc            # generate odoc HTML
```

Common dev loops:

```sh
dune build && dune runtest                # the usual cycle
dune exec app/server/bin/main.exe -- -port 12345 -seed-market-maker
dune exec app/client/bin/main.exe -- -port 12345 -name Alice
dune exec app/scenario_runner/bin/main.exe -- -scenario calm-day -port 12345 -seed 0
dune exec app/monitor/bin/main.exe -- -host localhost -port 12345
```

Never modify anything under `_build/` — it's regenerated by dune.

## Code conventions

Every file in this project follows these. Match the existing style; don't
introduce alternatives without a reason.

### Documentation

- Every lib needs docs
- Every module needs a comment
- All mli needs `(** doc *)`
- No useless comments (e.g., "adds numbers")
- Show examples
- Say how it fits with other modules
- Doc comments immediately after: `field : type (** doc *)`
- Use `[code]` and `{[blocks]}` in docs
- Use `{!Module.foo}` for links in docs
- Look at odoc output
- `(*_ *)`: ignored by doc tools

### Naming

- Short scope = short name
- Bools: `is_foo` not `check_foo`
- Can raise? End with `_exn`
- Grabs/frees stuff? Start with `with_`
- `with_`: try/finally pattern
- American English only
- `snake_case` not `camelCase`
- `_`: unused only
- `unsafe`: can segfault. Name it `unchecked` otherwise
- No negative bools (e.g., `dont_foo`)
- Name constants
- Break constants into pieces
- Type params: `'a 'b` unless special (`'ok 'err`, `'k 'v` for maps)

### Printing

- `[%string "x is %{x}"]` not `sprintf`
- Always use `sprintf !` for `ppx_custom_printf`
- Always derive `sexp_of`
- `[%message]` > `[%sexp]` for humans

### Testing

- Make readable
- Use expect tests
- Tests in separate dir
- Test-only stuff in `For_testing`
- Some inline tests just demonstrate
- Test files are named `test_<module>.ml` and live in `lib/<x>/test/` or
  `app/<x>/test/`.
- Tests are `let%expect_test "<name>" = ...` with `[%expect {| ... |}]`
  blocks. `let%test` is fine for property-style boolean checks; `let%test_unit`
  for tests without expected output. Look at `lib/order_book/test/` for the
  primary pattern.
- The shared test infrastructure is in `lib/test_harness/` — constants
  (`alice`, `bob`, `aapl`), order builders (`buy`, `sell`), and printers.
  Use them instead of building from scratch.
- When updating expect output, run `dune runtest --auto-promote` — but
  **read the diff first**. A surprising diff is a real signal.

### Interfaces

- Most modules have `type t`
- Most types are called `t`
- Args: `?optional`, `t`, positional, `~labeled`
- Label unclear args
- Optional args bad (except big APIs/tests)
- No new infix ops
- No `helpers.ml`
- Avoid functors (use first-class modules)

### Managing namespaces

- Only open if made for opening (`Let_syntax`, `O`, `Composition_infix`)
- Open `Async` consistently in lib (avoids interleaved print buffers)
- Use `O` for infix (or prefix ops if no `O`)
- `_intf.ml` for shared types
- Make top-level lib module
- Small lib = one module
- Sometimes export nothing
- Don't alias modules (if must: keep name same)
- Imports: choose per-file OR per-library consistently

### Style preferences

- Short match first
- Match > if
- No `else ()`
- No `let...and...` (except monads)
- Type annotations > module paths
- Module paths OK for: polymorphic functions or far from type annotation
- No `M.{ ... }` for records (incompatible with `[@@deriving fields ~getters]`)
- Normal variants > poly variants
- `f();` not `let () = f()`
- Pass `[%here]` when function takes `Source_code_position.t`
- `^/` for paths
- `Time_ns` > `Time_float`
- `flag_optional_with_default_doc_sexp`
- Inline complex helpers
- `Staged.t`: think before apply
- Sexp changes if variant type changes
- **`when` + mutable fields: segfault!** Simple conditions only

### Avoiding error-prone idioms

- No `| _ ->` when matching on variants
- Write types on ignored stuff (except record fields, labeled args, variant args)
- Use returned values
- No polymorphic compare
- Use `as` patterns for poly variants: `` (`A | `B) as mode -> mode ``

### Error handling

- Explicit error types
- No `exception` in interfaces
- Raise: `_exn` only
- Make `ok_exn` visible
- `with_`: get/free resource
- Don't eat user exceptions (use `Monitor.protect`)
- Check human input (`sexp`/`json`)
- Machine formats (`bin_io`): no validation needed
- Get ALL errors
- Add context
- For library-internal precondition violations: `raise_s [%message "..." (x : T.t)]`.
- For fallible operations exposed at module boundaries (especially RPCs):
  return `'a Or_error.t`, build errors with `Or_error.error_s [%message ...]`.
- For student-facing stubs (work you're meant to leave for the student):
  `failwith "TODO: implement <thing>"`. **Do not silently complete a TODO
  the student hasn't asked you to complete.**
- Do not use `Result.t` directly when `Or_error.t` will do — match the rest
  of the codebase.

### Monads

- `let%bind` > `>>=`
- One `Let_syntax` per scope (or use `let%bind.Module`)
- No pointless monads
- `;` before `return ()`
- `don't_wait_for` > `upon` (easier evolution)

### Opens

```ocaml
open! Core              (* always, for every src/test/bin .ml *)
open! Async             (* in libraries that use Async — gateway, bot_runtime *)
open Jsip_types         (* in libraries that use the core domain types *)
```

The `!` suppresses unused-open warnings. Don't replace `Core` with
`Stdlib`; don't import individual functions from `Core` (`open Core.List`
etc.).

### Dune files

Libraries follow a uniform pattern:

```
(library
 (name jsip_<x>)
 (public_name jsip-exchange.jsip_<x>)
 (libraries <deps>)
 (preprocess (pps ppx_jane)))
```

Tests:

```
(library
 (name jsip_<x>_test)
 (libraries jsip_<x> ... jsip_test_harness core)
 (inline_tests)
 (preprocess (pps ppx_jane)))
```

When adding a new library, mirror this — and check that the new directory
is reachable from the project (dune discovers libraries automatically as
long as they have a `dune` file).

all of week 1 and the first three week-2 exercises. **Before suggesting
an implementation, check whether the current code already has it.** If

  students build the `Slow_consumer` pathological bot.

## Stub conventions

Student-facing stubs use the shape:

```ocaml
let foo () = failwith "TODO: implement Foo.foo"
```

with a comment block above describing what the function should do and
why. **Don't fill these in unless the student explicitly asks you to.**
If a student asks "what does Foo.foo do?" and it's a stub, your answer
is: "it's a stub for them to implement; here's what it's meant to do
based on the comment and surrounding code."

## When the student asks you to…

- **"Explain this code"** — read the actual file. Don't summarize from
  the function name or the doc. Cite line numbers.
- **"Fix this bug"** — first ask what behavior is wrong vs. what's
  expected. The student should be able to state the bug; if they can't,
  help them get to a precise statement before changing code.
- **"Add a feature"** — check whether anything in the current code
  already does this or something close. Read existing similar features
  for reference. (For a new bot, the closest reference is
  `app/market_maker/src/market_maker.ml` — the dynamic market maker is
  the only fully-implemented `Bot_runtime.Bot` in the starter. For a
  new scenario, all four scenario stubs in `app/scenarios/src/` are
  still `failwith`; the building blocks are
  `Jsip_scenario_runner.Scenario_config.t` and `Bot_spec.t`.)
- **"Write tests for X"** — prefer adding to existing `test_X.ml` over
  creating new files. Use `Jsip_test_harness` helpers. Don't write tests
  that reimplement the code under test inside the test body — that's a
  tautological test and proves nothing.
- **"Plan how to do X"** — if the change touches more than one file,
  suggest using plan mode (`Shift+Tab`) so the student can review the
  plan before execution. Don't rush into edits.
